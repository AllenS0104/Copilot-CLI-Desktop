UI-V2 核心一句话

把“有用的东西”全部放到 Header + Sidebar + Main Area，
顶部 Menu 只负责“系统级善后工作”。


一、整体结构
你最终要稳定下来的桌面 UI 结构应该是：
┌──────────────────────────────────────┐
│ System Menu (File / View / Help)     │ ← 降权，不承载核心功能
├─────────────┬────────────────────────┤
│ Sidebar     │ Header (Context Bar)    │
│             ├────────────────────────┤
│ Projects    │                        │
│ Files       │  Main Workspace        │
│ Chats       │  (AI + Project)        │
│ Settings    │                        │
│             ├────────────────────────┤
│             │ Input / Actions        │
└─────────────┴────────────────────────┘

✅ 现代感来自这三点：

Menu 不再主导交互
Header 承载“当前上下文”
Sidebar 是“项目与能力入口”


二、Header

Header = “我现在在干什么 + 我可以马上做什么”

这是你现在 UI 最缺、但最值钱的一块。

1️⃣ Header 应该放什么（非常具体）
✅ 左侧：当前上下文（Context）
这是混合模式的灵魂
推荐结构：
📁 Project Alpha  ▾   /src/utils/logger.ts


项目名（可点击切换）
当前文件 / 当前作用域
如果没有项目：显示 No project selected

👉 这一步会让你的 App 从“聊天工具”升级成“工作台”

✅ 中间：可选（弱存在感）

搜索 / Command Palette 提示
或者留白（完全 OK）


✅ 右侧：轻量操作 + 状态
推荐从左到右：
⌘K  |  ● Connected  |  ⚙️


⌘K / Ctrl+K：Command Palette 入口
Connected 状态（可 hover 看模型 / workspace）
Settings（或用户头像）

❗注意：

Header 不应该堆按钮
所有“做事”的按钮，去 Main / Input


✅ Header 的设计原则（记住这 3 条）

不放任务型按钮
只显示“当前状态”
始终让用户知道：我在哪个项目里


三、Sidebar（从“图标栏”升级为“项目导航”）
你现在的 Sidebar 已经很接近正确答案了，只需要补一层信息结构。

2️⃣ Sidebar 的推荐分区
✅ 顶部：全局能力
💬 Chat
📁 Files
🕘 History


Chat：全局 / 项目对话
Files：项目文件浏览
History：会话 / 操作历史


✅ 中部：项目（核心）
Projects
▶ Project Alpha
▶ Project Beta


Project 是一级概念
点进去后，Main Area 内容发生变化


✅ 底部：系统级
⚙️ Settings


3️⃣ 收起 / 展开的正确策略

默认：图标 + 文字
提供 Collapse（你已经有了 ✅）
Collapse 后：

只留图标
hover 显示 tooltip



👉 这是 Linear / Slack / Notion 的成熟做法。

四、Main Area（真正“做事”的地方）
你现在这块已经做对方向了，我只帮你细化到产品级。

4️⃣ 空态（No Project / No Chat）
你现在的这个界面，其实就是理想的空态雏形 👍
我们只把它“职业化”一点。
✅ 标准空态结构
Start working with your project
Select a project or choose a task to get started.

[ Explain this project ]
[ Generate tests ]
[ Debug recent errors ]
[ Refactor code ]

✅ 每个卡片应该满足

一个动作动词
一个非常具体的承诺
点了之后一定发生事（哪怕是 placeholder）


5️⃣ 工作态（有项目 / 有对话）
Main Area 切换为：

上：AI 对话流
中：消息（支持 code block / file reference）
右（可选）：Context Panel（以后再做）


五、Input 区

在方案 里，输入区 = 主操作区


6️⃣ 输入区的推荐结构
[ + Files ] [ @Context ] [ /Commands ]
-------------------------------------
| Type a message...                | ▶
-------------------------------------

✅ 解释每一项存在的意义

+ Files：显式告诉用户“AI 能看文件”
@Context：强化“不是裸聊天”
/Commands：降低用户思考成本

👉 哪怕你一开始只是 UI 占位，也非常值得先做出来。

六、System Menu（你纠结的那一排）
现在回答你最关心的那点 👇
✅ 在方案 中，Menu 的正确定位是：

“你不想设计，但系统要求你有的东西”

✅ 推荐你直接这样处理：

File

Open Project
Close Project


View

Toggle Sidebar


Help

About



❗不放任何 AI / 项目内操作
这样你心理上会非常舒服，因为：

它“存在”
但它不抢戏
